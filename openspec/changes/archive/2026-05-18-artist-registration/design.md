## Context

The dandanna-api is a clean architecture Go API. All domain layers (entity, repository, service) are currently empty — this is the first domain feature. The `user` schema is defined in DBML but no migrations or Go code exist yet.

Key schema decisions already made:
- `user.user_management` and `user.artist_profile` use native PostgreSQL `uuid` PKs (`gen_random_uuid()`)
- Internal config/junction tables (`application`, `role`, `module`, `submodule`, `permission`, `user_role`) use `bigserial`
- All tables have audit fields (`created_by`, `updated_by`, `deleted_by`) and soft delete (`deleted_at`)
- `application` and `role` IDs are resolved at startup by querying `code`, not stored in env config

## Goals / Non-Goals

**Goals:**
- Introduce the full `user` schema via a single migration
- Seed the three portals and four roles required for the system to function
- Implement `POST /api/artists/register` following existing handler/service/repo patterns
- Establish the layer patterns (entity → repository interface → service → handler) for all future features

**Non-Goals:**
- Login / JWT issuance
- Artist profile update, delete, or listing
- APIs for managing `application`, `role`, `module`, `submodule`, or `permission` tables
- Email verification flow
- Client or backoffice user registration

## Decisions

### 1. Argon2id over bcrypt for password hashing
Argon2id won the Password Hashing Competition (2015) and is OWASP's first recommendation. It is memory-hard (64 MB), making GPU-based cracking significantly more expensive than bcrypt. Parameters: `time=1, memory=65536, threads=4, keyLen=32`. Same `golang.org/x/crypto` dependency, just `argon2` sub-package instead of `bcrypt`. Hash stored in PHC string format in a `TEXT` column.

### 2. Split user identity and artist profile into two tables
`user.user_management` holds identity (email, phone, password) for all user types. `user.artist_profile` holds artist-specific data. This enables a single login endpoint for all portals and allows one person to hold multiple roles without duplicate accounts — consistent with the `user_role` junction table design.

### 3. Three-INSERT atomic transaction
Registration creates `user_management` → `artist_profile` → `user_role` in a single `sqlx` transaction. If any INSERT fails (e.g. FK violation, unique constraint), the whole registration rolls back. UUID PKs are generated by PostgreSQL (`DEFAULT gen_random_uuid()`) and retrieved via `RETURNING id`.

### 4. Application and role IDs resolved at startup by code
`user.application` and `user.role` use `bigserial` PKs whose values differ per environment. Rather than storing IDs in env config (fragile), the app queries `SELECT id ... WHERE code = 'ARTIST_PORTAL'` at startup and fails fast if seed data is missing. This makes the dependency on seed data explicit and detectable before any request is served.

### 5. `bigserial` for internal config tables, `uuid` for user-facing tables
Internal tables (`application`, `role`, `module`, `submodule`, `permission`, `user_role`) are never exposed in API responses — `bigserial` is simpler, smaller, and faster for joins. User-facing tables (`user_management`, `artist_profile`) use `uuid` to prevent sequential ID enumeration.

### 6. Email uniqueness checked at application layer before DB write
Rather than catching a PostgreSQL unique constraint violation and mapping it to a 409, the service calls `ExistsByEmail` first. This produces a clean `*domainerror.Conflict` with a meaningful message rather than parsing a DB error string.

## Risks / Trade-offs

- **Seed data dependency** → Mitigated by fail-fast startup query; the server won't start if seed is missing
- **Argon2id memory cost (64 MB per hash)** → Under concurrent load, registration could spike memory; acceptable at this scale, revisit if registration volume grows significantly
- **No email verification** → An artist with a typo in their email cannot recover without a reset flow; acceptable for MVP, must be added before production launch
- **`updated_at` requires a trigger** → PostgreSQL doesn't auto-update `updated_at`; a trigger is included in `001_create_user_schema.sql` to handle this at the DB level rather than relying on application code

## Migration Plan

1. Run `migrations/001_create_user_schema.sql` — creates full `user` schema and `updated_at` trigger
2. Run `migrations/002_seed_user_schema.sql` — inserts 3 applications and 4 roles
3. Deploy application — startup queries resolve `ARTIST_PORTAL` app ID and `ARTIST` role ID; server starts
4. Rollback: drop `user` schema and all its objects (no other schemas depend on it yet)
